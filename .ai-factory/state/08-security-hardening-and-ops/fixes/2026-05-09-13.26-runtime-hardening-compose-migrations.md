# Fix Trace: runtime-hardening Compose migrations

**Date:** 2026-05-09 13:26
**Change:** `08-security-hardening-and-ops`
**Source:** current pointer

## Finding

`GET /portal` through the Compose nginx runtime could reach PHP-FPM before Laravel migrations were applied. The request then used the session guard remember-me path, queried `users`, and failed against an empty PostgreSQL schema.

Observed evidence:

- PostgreSQL: `relation "users" does not exist`
- Laravel: query from `Illuminate\Auth\EloquentUserProvider`
- nginx: FastCGI upstream timeout while waiting for `/portal`
- PHP-FPM notices about ignored `user`/`group` pool directives while running non-root

## Root Cause

`compose.yml` waited only for PostgreSQL socket readiness and used an app healthcheck that ran `php -v`. That proved PHP was installed, but did not prove Laravel's database schema was present. The entrypoint also dropped from root to `www-data` via a shell string, and default PHP-FPM pool `user`/`group` directives remained active even though PHP-FPM was already launched as a non-root user.

## Fix

- Made entrypoint migrations mandatory before PHP-FPM starts.
- Changed privilege dropping to `su-exec www-data "$@"` to preserve argument boundaries.
- Commented default PHP-FPM pool `user`/`group` directives during image build to avoid rootless FPM notices.
- Changed the app healthcheck to require `php artisan migrate:status`.
- Changed nginx to wait for app `service_healthy`.
- Updated README Docker startup instructions.
- Added `tests/Unit/RuntimeHardeningTest.php`.
