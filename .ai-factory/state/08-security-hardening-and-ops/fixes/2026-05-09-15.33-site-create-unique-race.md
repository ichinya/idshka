# Fix Trace: Site Create Unique Race

Date: 2026-05-09 15:33:03 +05:00
Change-id: `08-security-hardening-and-ops`
Mode: OpenSpec-native mode

## Selected Finding

Local browser flow hit a Laravel 500 while submitting a portal site for `https://localhost:8443/`:

- `Illuminate\Database\UniqueConstraintViolationException`
- PostgreSQL constraint: `sites_owner_domain_unique`
- Duplicate key: `(owner_user_id, normalized_domain)=(1, localhost)`

The nginx logs showed an earlier `POST /portal/sites` returning `499`, which can leave the user retrying while the first request still created the row.

## Root Cause

`CreateSiteAction` had an application-level pre-check for an existing owned domain, but the database unique index is still the final concurrency boundary. A second request can pass the pre-check before another request's insert is visible, then fail at the `sites_owner_domain_unique` insert. That database-level conflict was not translated back into the domain `SiteDomainConflictException`, so the portal rendered a 500 instead of a normal form error.

## Fix

- Catch `UniqueConstraintViolationException` around the site creation transaction.
- Only translate known owner/domain unique conflicts (`sites_owner_domain_unique` on PostgreSQL, equivalent SQLite test message) into `SiteDomainConflictException`.
- Log `[site.create] duplicate_domain_for_owner_after_unique_race` with safe owner/domain/site identifiers.
- Keep unrelated unique violations rethrown so primary-key or other integrity bugs are not hidden.
- Added a portal regression test that inserts a duplicate site between the pre-check and insert to reproduce the race deterministically.
