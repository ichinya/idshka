# Add rollback signing-key command

**Date:** 2026-05-03 21:56
**Change ID:** 08-security-hardening-and-ops
**Resolver source:** current-pointer
**Files:** routes/console.php, tests/Feature/SigningKeyCommandTest.php
**Severity:** high

## Selected Finding

Code review found that the active OpenSpec change required an executable issuer signing-key rollback operation, but the operator command namespace exposed only status, prepare, activate-next, retire-expired, and force-retire commands.

## Root Cause

`SigningKeyService::rollbackToPreviousActive()` already implemented the lifecycle operation, and service-level tests covered it, but the operator-facing Artisan command layer was not updated. The OpenSpec requirement covers executable operator operations, so service coverage alone was not enough.

## Solution

Added `idshka:keys:rollback` in `routes/console.php` with:

- `--dry-run` to show the newest active key that would be retired and the previous active key that would continue signing.
- `--force` confirmation for the destructive path.
- Safe structured command logs using `kid`, database key ids, status, and error class only.
- `SigningKeyStateException` handling when fewer than two active signing keys are available.

Added a feature regression test proving dry-run behavior, force refusal, successful rollback, and active signing-key selection after rollback.

## Prevention

- Code review for OpenSpec operator requirements must check both service APIs and executable operator entrypoints.
- Command-level tests should cover every lifecycle method that is documented as operator-executable.
- Verification should inspect `php artisan list <namespace>` when a finding is about missing CLI surface.
