# Fix Trace: Production runtime migrations

**Date:** 2026-05-09 23:25
**Change ID:** 09-portal-frontend-redesign
**Source:** `$aif-fix` after `$aif-review` blocker

## Finding

Production entrypoint used `php artisan migrate` without `--force`. Laravel cancels that command in headless `APP_ENV=production`, so pending migrations would not apply.

## Fix

- Added `idshka:runtime-migrate` in `routes/console.php`.
- Added production pretend preflight that blocks destructive pending migration SQL.
- Updated `infra/docker/php/entrypoint.sh` production branch to call the runtime migration command.
- Updated `tests/Unit/RuntimeHardeningTest.php` to reject the broken bare production migrate path.
- Updated `README.md` runtime wording.

- `docker compose --profile examples exec -T app sh -lc 'APP_ENV=production DB_DATABASE=idshka_runtime_migrate_smoke php artisan idshka:runtime-migrate --no-interaction; printf "exit:%s\n" "$?"'`: applied 15 migrations on an empty temporary database, exit `0`.
- `docker compose --profile examples run --rm -e APP_ENV=production app php -v`: entrypoint ran `[FIX:runtime-migrate]` and completed with `migrations_count=0`.
