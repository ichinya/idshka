## Problem

`aif-review` found three issues:

1. Token revoke loaded `ApiToken` by global id before ownership scoping, so foreign existing ids returned `403` while missing ids returned `404`.
2. Gateway accepted JWT `aud` as an array but forwarded `tostring(claims.aud)`, producing a Lua table address in `X-Idshka-Audience`.
3. Gateway docs listed `X-Idshka-Email` and `X-Idshka-Roles` as current runtime headers even though the reference gateway does not emit them.

## Root Cause

- Revoke ownership was enforced in `RevocationService` after a global lookup, creating an observable difference between foreign and missing tokens.
- `jwt_validate.lua` used `audience_matches` only as a boolean check and discarded the matched configured audience before trusted header injection.
- The gateway contract mixed current reference-runtime headers with future optional identity/role headers.

## Solution

- Scoped revoke lookup by `id` and authenticated `user_id`, constrained the route to numeric ids, and returned `404 token_not_found` for missing and foreign tokens.
- Added a regression feature test proving foreign token ids are not revealed.
- Preserved the matched configured audience in `jwt_validate.lua` and used it when building `X-Idshka-Audience`.
- Extended gateway smoke with an array-audience token case and updated the local smoke token helper to generate multiple `aud` values from comma-separated input.
- Split docs into current emitted headers and future optional headers.

## Prevention

- Owner-bound endpoints should scope lookup queries before branching on not-found vs forbidden responses.
- Gateway context headers should be derived from normalized validation output, not directly from raw JWT claim shapes.
- Docs should distinguish implemented reference behavior from future optional contract extensions.
