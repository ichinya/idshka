# Gateway header namespace sanitization

**Date:** 2026-04-26 15:05
**Files:** infra/openresty/demo-resource/lua/context_headers.lua, infra/openresty/demo-resource/smoke.sh, examples/demo-resource-api/nginx.conf
**Severity:** high

## Problem

GitHub review found that the gateway did not strip the full client-supplied `X-Idshka-*` namespace before proxying to the upstream. A spoofed header such as `X-Idshka-Canary` could reach the API resource even though trusted context headers are supposed to be gateway-owned.

## Root Cause

The prefix check used `string.sub(string.lower(name), 1, 9) == "x-idshka"`. For a namespaced header like `X-Idshka-User-Id`, the first nine characters are `x-idshka-`, so the comparison failed. Existing smoke coverage sent spoofed headers that were later overwritten by trusted values, which masked the sanitizer bug.

## Solution

Replaced the magic substring comparison with explicit constants for `x-idshka` and `x-idshka-`. Added smoke coverage with a canary `X-Idshka-Canary` header and exposed that header in the smoke upstream response so arbitrary namespaced header leakage is detected.

## Prevention

- Avoid magic lengths in security boundary prefix checks; compare against named prefix constants.
- Smoke tests for header sanitization must include at least one arbitrary namespaced header that is not overwritten later.
- Code review should check both trusted-header replacement and removal of unknown trusted-namespace inputs.
