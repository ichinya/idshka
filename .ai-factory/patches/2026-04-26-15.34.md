# OAuth Web Login Review Findings

Date: 2026-04-26 15:34
Severity: high
Tags: #oauth #oidc #pkce #privacy #security #review-fix #laravel

## Problem

Code review found three issues in the web login flow:

- ID tokens always included `name` and `email`, even when the client requested only `openid`.
- The authorize callback appended `?code=...` directly and broke registered redirect URIs that already contained a query string.
- PKCE request validation allowed weak verifier and challenge strings instead of enforcing RFC-style length and character constraints.

## Root Cause

The ID token builder copied user PII without checking requested OIDC scopes, redirect URL construction assumed a queryless callback URI, and PKCE shape validation was deferred to hash comparison instead of being enforced at the request and domain-service boundaries.

## Solution

- Scope-gated ID token PII claims: `name` is emitted only for `profile`, and `email` only for `email`.
- Added redirect response URL construction that preserves an existing query string and appends `code`/`state` using the correct separator.
- Added PKCE verifier and S256 challenge format constants, request validation rules, and defensive checks in `PkceService`.
- Added regression tests for `scope=openid`, redirect URIs with existing query parameters, malformed PKCE challenges, and weak PKCE verifiers.

## Prevention

Keep privacy-sensitive token claims explicitly tied to scopes, cover callback URL construction with query-bearing redirect URIs, and validate protocol syntax before persisting or exchanging OAuth/PKCE values.
