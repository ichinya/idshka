# Production runtime migrations cancelled by Laravel guard

**Date:** 2026-05-09 23:25
**Files:** infra/docker/php/entrypoint.sh, routes/console.php, tests/Unit/RuntimeHardeningTest.php, README.md
**Severity:** high

## Problem

The Docker entrypoint tried to satisfy "no production --force" by running `php artisan migrate` directly when `APP_ENV=production`.
Laravel cancels that command in headless production runtimes before applying pending migrations, so the container could start only if the schema was already current.

## Root Cause

Laravel's stock `migrate` command uses `ConfirmableTrait` in production. In a non-interactive container, that confirmation resolves to false unless the command is run with `--force`.
Removing `--force` from the entrypoint therefore preserved the text of the policy but made migration execution non-functional.

## Solution

Added `idshka:runtime-migrate`, a project runtime command that calls Laravel's `Migrator` service directly and logs `[FIX:runtime-migrate]` start/completion metadata.
In production it first runs a pretend preflight and blocks pending destructive SQL such as `DROP TABLE`, `ALTER TABLE ... DROP COLUMN`, `TRUNCATE`, or `DELETE FROM`, while allowing non-destructive constraint relaxations such as `DROP NOT NULL`.
The production entrypoint now calls `php artisan idshka:runtime-migrate --no-interaction`, while local runtimes still call the stock `php artisan migrate --no-interaction`.
The README and runtime hardening tests were updated to lock in the working production path without `php artisan migrate --force`.

## Prevention

- Review production migration changes against actual Laravel command behavior, not only string-level policy.
- For container entrypoints, verify the exact command in an `APP_ENV=production` container.
- Keep regression coverage that rejects both `php artisan migrate --force` and a bare production `php artisan migrate` call.
- Keep production migration preflight fail-closed for destructive SQL before applying pending migrations.
