# Review findings: issuer revoke enumeration and gateway audience headers

**Date:** 2026-04-26 10:05
**Files:** `app/Http/Controllers/Api/Issuer/RevokeUserApiTokenController.php`, `routes/api.php`, `infra/openresty/demo-resource/lua/jwt_validate.lua`, `infra/openresty/demo-resource/lua/context_headers.lua`, `infra/openresty/demo-resource/smoke.sh`, `routes/console.php`, `tests/Feature/IssuerApiFlowTest.php`, `docs/GATEWAY_CONTRACT.md`
**Severity:** medium

## Problem

Review found that revoke token ids were enumerable, array-form JWT audiences were not normalized before being forwarded as trusted gateway context, and gateway docs listed headers not emitted by the runtime.

## Root Cause

The revoke controller performed a global `ApiToken` lookup and deferred ownership rejection to the service layer, creating different responses for foreign vs missing ids. The gateway validator only returned raw claims after audience validation, so `context_headers.lua` had no normalized audience value and converted Lua tables directly to strings. The docs were written as a target contract and not narrowed to the implemented reference header set.

## Solution

Scoped the revoke query by authenticated user before deciding `404`, constrained the route id to numbers, added a feature regression for foreign revoke ids, carried the matched audience through JWT validation, normalized `X-Idshka-Audience`, added an array-audience smoke case, taught the local smoke token helper comma-separated audiences, and split current vs future gateway headers in docs.

## Prevention

Owner-sensitive lookup endpoints should query in owner scope first and use indistinguishable missing/foreign responses unless a product requirement says otherwise. Gateway validation should expose normalized context fields to header mapping code. Documentation should explicitly label target/future headers when runtime coverage does not exist yet.
