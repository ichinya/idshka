# Site create unique race returned 500

**Date:** 2026-05-09 15:33
**Files:** `app/Domain/Sites/Actions/CreateSiteAction.php`, `tests/Feature/PortalManagementFlowTest.php`
**Severity:** medium

## Problem

Retrying the portal site creation flow for `https://localhost:8443/` could surface a Laravel 500 when a previous request had already created the same `(owner_user_id, normalized_domain)` row.

## Root Cause

The action checked for an existing owned site before inserting, but the database unique index is the real concurrency boundary. A duplicate insert raised `UniqueConstraintViolationException` from the database and bypassed the existing `SiteDomainConflictException` handling used by the portal and API controllers.

## Solution

`CreateSiteAction` now catches owner/domain unique violations from the transaction and translates them to `SiteDomainConflictException` with safe warning logging. The catch is scoped to the known `sites_owner_domain_unique` conflict so unrelated unique failures still fail loudly.

## Prevention

- Treat database unique constraints as concurrency boundaries even when application pre-checks exist.
- Regression-test race windows by inserting conflicting rows between pre-check and write.
- Keep unique-violation mapping narrow so operational form/API conflicts do not mask data integrity bugs.
