# Portal guest auth route failed after Docker startup

**Date:** 2026-05-02 08:52
**Files:** routes/web.php, app/Http/Controllers/Auth/LoginController.php, app/Http/Controllers/Auth/RegisterUserController.php, app/Http/Controllers/Auth/LogoutController.php, resources/views/auth/login.blade.php, tests/Feature/AuthSocialiteFlowTest.php
**Severity:** medium

## Problem

Docker Compose could build and start the Laravel stack, but opening the protected `/portal` page as a guest produced a Laravel `500` with `Route [login] not defined`. This made the local Docker runtime look broken even though containers were healthy.

## Root Cause

The portal route uses Laravel's `auth:web` middleware. For unauthenticated browser requests, Laravel redirects guests to a route named `login`. The project only had JSON-oriented `POST /login` named `auth.login`; there was no browser `GET /login` route named `login`.

## Solution

Added browser `GET /login` and `GET /register` routes backed by an auth view. The existing JSON auth endpoints remain named `auth.login`, `auth.register`, and `auth.logout`; controllers now return redirects for regular browser form requests while preserving JSON responses for requests that expect JSON.

## Prevention

- When adding browser-protected routes with `auth:web`, ensure a named `login` GET route exists.
- Keep separate tests for browser auth redirects and JSON auth API responses.
- Diagnose Docker startup by checking container health and HTTP logs before changing runtime images such as PHP-FPM or FrankenPHP.
