# Harden site registry security boundaries and verification safety

**Date:** 2026-04-22 23:09
**Files:** routes/api.php, app/Providers/AppServiceProvider.php, app/Domain/Sites/Actions/VerifySiteDomainAction.php, app/Domain/Sites/Services/WellKnownFileVerificationChecker.php
**Severity:** high

## Problem

Site registry endpoints accepted state-changing requests without explicit route-level policy checks and without dedicated rate limits. Domain verification also had a race window where competing owners could verify the same normalized domain, and file verification allowed outbound checks without private/reserved IP filtering.

## Root Cause

Security controls were partially implicit:
- ownership checks lived in controller logic instead of policy middleware;
- no dedicated limiter existed for the new site-registry write endpoints;
- verification success path did not re-check verified-domain ownership under a row lock;
- outbound file verification trusted DNS resolution and redirects too broadly.

## Solution

Applied defense-in-depth changes:
- moved site routes to explicit middleware stack: `web`, `auth:web`, `throttle:site-registry`;
- enforced `can:manage,site` policy middleware for verify/mode endpoints;
- added `RateLimiter::for('site-registry', perMinute(30))` keyed by user id or IP;
- added transactional conflict guard in verification flow with `lockForUpdate` and fail-closed error `verified_domain_owned_by_another_user`;
- hardened file verification with public-IP-only DNS resolution check and `withoutRedirecting()`;
- added `[FIX:security]` logs for blocked conflict/SSRF paths.

## Prevention

- For new write endpoints, require explicit guard + policy + throttle at route declaration time.
- For ownership claims, repeat critical integrity checks inside DB transactions with locking.
- Treat all outbound verification fetches as SSRF surfaces and block private/reserved targets.
- Keep targeted feature tests for guarded and failure paths before merge.
