# Closed probe trust drift and CI ingress blind spot

**Date:** 2026-04-20 15:11
**Files:** `.github/workflows/ci.yml`, `app/Http/Middleware/SetSecurityHeaders.php`, `routes/probes.php`, `tests/Feature/FoundationSmokeTest.php`
**Severity:** medium

## Problem

Two review findings remained after the previous security hardening pass:

1. The public foundation metadata endpoint built absolute probe URLs from the incoming request host, so a spoofed `Host` header could make `/` advertise attacker-controlled origins.
2. CI verified Laravel tests and `docker compose config`, but never exercised the real public nginx ingress, so the `/ready` edge protection could regress without any automated signal.

## Root Cause

The application still consumed untrusted request metadata in places that looked operationally harmless but were still part of a public contract. The root endpoint used `url(...)`, which depends on request host/scheme context, and the security-header middleware also trusted raw `X-Forwarded-Proto` outside Laravel's proxy trust model. Separately, the project relied on manual live verification for ingress-only behavior because the workflow never booted the Compose stack and hit the browser-facing edge.

## Solution

The probe contract was changed to advertise relative named routes (`/health`, `/up`) instead of absolute URLs, and the root probe log now records the sanitized relative paths with the `[FIX:probe-trust]` prefix. HSTS emission was tightened to rely only on `$request->isSecure()`, which means forwarded proto is honored only after proper trusted-proxy configuration exists. Regression coverage was expanded with a test that proves spoofed `X-Forwarded-Proto` does not add HSTS on plain HTTP. CI now starts the Compose stack and runs ingress smoke checks for `200 /health`, `403 /ready`, and `200 /healthz` on the gateway, plus logs/cleanup on failure.

## Prevention

- Do not expose absolute URLs in low-level operational metadata unless the host/proxy trust boundary is explicitly configured.
- Treat raw forwarding headers as untrusted input until the framework proxy trust chain is configured; security headers should follow framework trust decisions, not header strings.
- When a security control lives at nginx/OpenResty edge, add a runtime smoke check in CI instead of relying only on in-process framework tests.
- Keep probe contracts host-agnostic and simple so they cannot accidentally become redirect/issuer metadata later.
