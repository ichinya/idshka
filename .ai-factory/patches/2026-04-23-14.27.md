# Auth Socialite feature tests exposed string normalization crash

**Date:** 2026-04-23 14:27
**Files:** `tests/Feature/AuthSocialiteFlowTest.php`, `tests/Feature/FoundationSmokeTest.php`, `app/Domain/Identity/Services/SocialiteProfileFactory.php`, `app/Domain/Identity/Actions/HandleSocialCallbackAction.php`, `app/Http/Middleware/AssignRequestId.php`, `.ai-factory/plans/02-user-auth-socialite/status.yaml`, `.ai-factory/plans/02-user-auth-socialite/fixes/2026-04-23-14.27-auth-socialite-feature-tests.md`
**Severity:** medium

## Problem

Verification for `02-user-auth-socialite` failed because public auth and Socialite endpoints had no targeted feature/smoke tests. After adding those tests, mocked Socialite callback requests returned `500` with:

```text
Call to a member function toString() on string
```

## Root Cause

The Socialite implementation used `Str::limit(...)->toString()` in paths that receive plain strings. In the current Laravel version, `Str::limit()` returns a plain string, not a `Stringable`, so the callback crashed before user/social-account persistence. Because there were no Socialite feature tests, the bug was not caught by the existing foundation and site-registry suites.

The same incorrect assumption existed in request-id sanitization and would have failed for requests carrying a custom `X-Request-Id` header.

## Solution

Added `AuthSocialiteFlowTest` covering:

- register/login/logout session auth;
- mocked Socialite redirect and callback;
- repeated Socialite callback resolving the existing user;
- account link/unlink;
- unsupported provider errors;
- provider token encryption and log-safety expectations.

Replaced `Str::limit(...)->toString()` with direct `Str::limit(...)` string usage in Socialite profile normalization, generated Socialite user names, and request-id sanitization.

Added request-id sanitization regression coverage so the non-Socialite instance of the same helper-return-type bug is also covered.

## Prevention

- Public auth endpoints must land with feature tests in the same change.
- Mock external OAuth providers in tests so callback persistence, repeat login, and token handling are exercised without real provider calls.
- Treat framework helper return types as version-specific; do not chain `Stringable` methods unless the code uses `Str::of(...)`.
- Include at least one token log-safety test for provider-token flows.
