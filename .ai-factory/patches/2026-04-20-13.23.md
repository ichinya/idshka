# Closed readiness probe reverse-proxy bypass

**Date:** 2026-04-20 13:23
**Files:** `README.md`, `app/Http/Middleware/EnsureInternalProbeAccess.php`, `infra/docker/nginx/default.conf`, `tests/Feature/FoundationSmokeTest.php`
**Severity:** high

## Problem

The previous readiness hardening looked correct in Laravel feature tests but still exposed `/ready` through the public nginx ingress. Live requests to `http://localhost:8080/ready` returned `200 OK`, and adding `X-Forwarded-For: 203.0.113.10` did not change that result.

## Root Cause

The access decision lived only inside the Laravel app and relied on app-visible client IP. Behind nginx, that IP reflects the proxy hop or a header chain, not the ingress policy itself. In other words, the code tried to secure a public probe at the application layer after the request had already passed the public edge. The existing regression test only mutated `REMOTE_ADDR` in a direct app test, so it missed the reverse-proxy path entirely.

## Solution

The public nginx ingress now fails closed for `/ready` and returns a hardened `403` JSON response directly, so the readiness probe is no longer exposed on the browser-facing surface. The Laravel middleware remains as defense in depth for non-nginx/internal paths, now resolves the original client IP from `X-Forwarded-For` when present, and emits `[FIX:ready-access]` allow/deny logs with request and transport context. Regression coverage was expanded with a proxy-style test case that simulates a private proxy hop carrying a public forwarded client IP.

## Prevention

- Do not rely on app-only IP checks to secure endpoints that should not exist on the public ingress in the first place.
- Put ingress restrictions at the edge layer first, then keep application checks as a second barrier.
- When fixing proxy-sensitive security issues, verify both direct app tests and the real reverse-proxy path with live curls.
- Add at least one regression test that models the proxy hop explicitly, not just a single `REMOTE_ADDR` override.
