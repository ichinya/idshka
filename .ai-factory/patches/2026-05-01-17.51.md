# OpenSpec plan 6 validation and authorize middleware fix

**Date:** 2026-05-01 17:51
**Files:** `openspec/changes/06-web-login-through-idshka/*`, `routes/oauth.php`, `bootstrap/app.php`, `tests/Feature/OAuthWebLoginFlowTest.php`, `.ai-factory/ARCHITECTURE.md`
**Severity:** medium

## Problem

Plan `06-web-login-through-idshka` was implemented in code, but the OpenSpec change was not valid because it had no `specs/**/spec.md` deltas. The authorize route also did not include the planned `auth:web` middleware, and `git diff --check` reported trailing whitespace in `.ai-factory/ARCHITECTURE.md`.

## Root Cause

The legacy plan migration preserved prose artifacts (`proposal.md`, `design.md`, `tasks.md`) but did not create OpenSpec-native deltas or checkbox tasks. The OAuth authorize endpoint relied on a defensive controller-level guest check instead of the route middleware required by the plan, so adding `auth:web` changed unauthenticated JSON responses to Laravel's default shape until the exception handler was adjusted.

## Solution

Added an `oauth-web-login` OpenSpec delta with requirements and scenarios, converted the plan 6 proposal to `## Why` / `## What Changes`, and added a canonical checkbox task list so `openspec list` reports the change as complete. Added `auth:web` to `GET /oauth/authorize`, mapped unauthenticated authorize requests back to the deterministic OAuth error shape, added a route middleware regression test, and removed trailing whitespace from `.ai-factory/ARCHITECTURE.md`.

## Prevention

- Migrated OpenSpec changes need `specs/<capability>/spec.md` deltas before verification or archive.
- `tasks.md` should include `- [x]` / `- [ ]` checkbox tasks if `openspec list` task status matters.
- Verify route middleware with `php artisan route:list --path=oauth -vv` when a plan requires exact middleware.
- On Windows, prefer `php vendor/bin/pint ...` over invoking the `vendor/bin/pint` shim directly from PowerShell.
