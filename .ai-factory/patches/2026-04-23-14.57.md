# Harden Socialite linking and identity validation

**Date:** 2026-04-23 14:57
**Files:** app/Http/Controllers/Auth/SocialiteCallbackController.php, app/Http/Requests/Auth/RegisterRequest.php, app/Domain/Identity/Services/SocialiteProfileFactory.php, tests/Feature/AuthSocialiteFlowTest.php
**Severity:** high

## Problem

Review found three auth-flow defects in the Socialite/public auth work:

- Socialite account linking callbacks trusted a session-stored `user_id` without requiring the same user to be actively authenticated.
- Registration validated `unique:users,email` before lowercasing email for persistence, allowing case variants to bypass validation and hit the database unique constraint.
- Provider identity keys were truncated before being used as the principal social account lookup key.

## Root Cause

The implementation separated redirect-time context creation from callback-time authorization too loosely. The callback treated the session intent as enough authority for linking, even though link actions must be bound to an active first-party Laravel session.

Email normalization lived in the controller after FormRequest validation, so validation and persistence did not operate on the same canonical value.

The profile factory reused display-field truncation behavior for `provider_user_id`, but identifiers must not be lossy because they are security-sensitive lookup keys.

## Solution

Added regression coverage for stale link intents, mismatched authenticated users, case-variant registration, and overlong provider IDs.

Changed link callback resolution to require `$request->user()` and require its id to match the stored linking intent before contacting the external provider. Missing or mismatched link auth now fails closed with `401 linking_auth_required` and logs non-secret context.

Moved registration email normalization into `RegisterRequest::prepareForValidation()` so the unique rule sees the persisted canonical form.

Changed provider ID handling to reject values longer than the `social_accounts.provider_user_id` column limit instead of truncating them.

## Prevention

- Treat session intent as state, not authorization, for auth-sensitive callbacks.
- Normalize request data before validation when controllers persist normalized values.
- Never truncate principal identifiers used for authentication, linking, or unique lookup.
- Add regression tests for stale session, mismatched user, canonicalized uniqueness, and identifier boundary length.
