# Gateway smoke TTL mismatch with running gateway container

**Date:** 2026-05-03 22:30
**Files:** `infra/openresty/demo-resource/smoke.sh`, `tests/Unit/CiWorkflowSecurityTest.php`
**Severity:** high

## Problem

Gateway smoke could fail `stale key after ttl expiry` with HTTP 200 when the gateway was started with `GATEWAY_JWKS_CACHE_SECONDS=30`. The smoke script waited its own default TTL instead of the running gateway container's actual TTL, so the assertion could execute before the cached key was supposed to expire.

## Root Cause

The smoke script assumed the shell running the script always had the same `GATEWAY_JWKS_CACHE_SECONDS` environment as the `docker compose` invocation that created the gateway container. That is not always true across host shells and Windows/Unix process boundaries. Compose can pass a TTL to the container while the script falls back to `20`, causing a false stale-key failure for gateway TTL values greater than the script fallback.

## Solution

The smoke script now resolves the cache wait from explicit `JWKS_CACHE_SECONDS`, then shell `GATEWAY_JWKS_CACHE_SECONDS`, then the running `demo-resource-gateway` container's `GATEWAY_JWKS_CACHE_SECONDS`, and only then falls back to `20`. It validates the resolved value and logs the non-secret TTL decision with `[FIX:gateway-smoke-cache-ttl]`.

## Prevention

- For Docker smoke scripts, derive assertion timing from the running container when the script validates runtime behavior.
- Treat host-shell environment and Compose interpolation as separate boundaries during review.
- Keep regression coverage that checks the smoke script can read runtime configuration, not just hard-coded defaults.
