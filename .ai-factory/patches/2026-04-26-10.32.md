# Issuer revoke cache fallback and revocable token id

**Date:** 2026-04-26 10:32
**Files:** app/Domain/Issuer/Services/RevocationService.php, app/Domain/Issuer/Actions/IssueUserApiTokenAction.php, app/Domain/Issuer/DTO/IssuedUserApiToken.php, app/Http/Controllers/Api/Issuer/IssueUserApiTokenController.php, tests/Feature/IssuerApiFlowTest.php, tests/Unit/Issuer/RevocationServiceTest.php, docs/API_FLOWS.md, docs/GATEWAY_CONTRACT.md
**Severity:** medium

## Problem

Review found two issuer lifecycle gaps. First, token revoke committed the database changes before writing to the optional Redis denylist, but a cache backend failure still bubbled out as a 500 response. Second, the issue endpoint returned a raw token and metadata but not the `api_tokens.id` needed by the public revoke route.

## Root Cause

The implementation treated the Redis denylist cache as part of the critical revoke path even though project docs define the database as source of truth and Redis as an accelerator. The issue flow also created the `api_tokens` row as a side effect without carrying the created row id back through the DTO and controller response.

## Solution

Wrapped denylist cache writes in a best-effort `try/catch` with `[FIX:issuer-revoke-denylist-best-effort]` warning logs after the database revoke has been persisted. Added `tokenId` to the issued token DTO returned by the action, included `api_token_id` in completion logs, and exposed `token_id` in the issue response. Updated docs to describe `token_id` and the best-effort cache contract.

## Prevention

- Review optional cache integrations for post-commit exception handling.
- Ensure every public lifecycle endpoint returns the identifier required by its matching follow-up endpoint.
- Add regression tests that force cache failures and use only public API response data for follow-up actions.
